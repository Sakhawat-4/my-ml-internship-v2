FlyRank ML Internship — Capstone, Lane 2: Refresh Opportunity Scoring

Which pages get reviewed first?

A ranking model for content-refresh review — and an honest look at what changes when the validation split stops leaking.

Author Sakhawat Repo Sakhawat-4/my-ml-internship-v2 Data credit FlyRank

Question. Out of thousands of content pages, which ones should a reviewer with limited time look at first?

Data. An anonymized slice of FlyRank search-performance data — 30,000 pages across 32 clients, 52 engineered features, no titles, URLs, client names, or keywords used.

Method. A Random Forest ranker trained against a transparent rule-based baseline, validated on a client-grouped holdout — the split a reviewer actually faces, since a brand-new client's pages should never leak into training.

Result. Measured honestly, the model roughly doubles the baseline's precision at the top of the queue (p@50: 0.74 vs 0.36) — and a naive random split would have overstated that same comparison by a wide margin, which turned out to be a finding in its own right.

What it's for. A ranked, reason-coded queue that sits in front of a human editor — a reviewer aid, not an auto-publish decision.

01 — Introduction

The decision this supports

The unit of analysis is one content page, for one client. The decision it feeds: given a reviewer who can only manually check a limited number of pages each week, which ones are worth their time first?

The action on the other end is concrete — a reviewer opens the top-ranked pages and decides whether to refresh, expand, protect, prune, or simply keep monitoring. Getting the ranking wrong costs real time in both directions: a false flag burns a reviewer's limited attention on a page that didn't need it, while a missed decliner keeps losing search visibility unnoticed until the next audit.

A hand-written rule — "flag anything with high impressions and no update in 180 days" — treats every signal as equally important and misses how they interact. The starter pipeline's own baseline already shows a learned ranking finding meaningfully more of the true decliners than that kind of fixed rule, which is the working hypothesis this capstone tests properly, on a split that can't quietly cheat.

What went in, and what was left out on purpose

The source is FlyRank's bundled, anonymized starter release. The label — is_declining_label — has a base rate of 0.542: slightly more than half the pages in this sample are already flagged as declining, which matters, because any accuracy or precision number below is only meaningful measured against that base rate, not against zero.

Deliberately excluded: titles, URLs, client names, domains, and search keywords. Nothing client-identifying appears anywhere in this repo or in the page you're reading. Client IDs are used only as a grouping key for the validation split — never as a feature the model can learn from directly.

03 — Methodology

Baseline, model, and the split that almost lied

The baseline is a transparent, hand-readable rule built in Week 4: a small set of threshold checks on traffic, position, and freshness signals, each contributing a reason code — no black box, nothing a reviewer couldn't sanity-check by eye.

Why the split gets its own paragraph

A first-pass, row-level random split put 31 of the 32 clients in both train and test at once — meaning the model could partly learn "what client 3fdba35f0 tends to look like," not just what a declining page looks like in general. That's a leak, and it's the exact failure mode a client-grouped split exists to close: the 20% holdout used for every honest number below contains clients the model never saw during training.

A second, independent check ran the full leakage-hunt checklist against the final feature set: no label-derived or ID column made it into the model's inputs, the single highest raw correlation with the label across all 52 features was 0.190 (nowhere near the "one feature towers over the rest" pattern that signals a hidden leak), and — as a self-test of the test itself — deliberately injecting the actual leak column pushed the score to a suspicious 1.00 AUC, confirming the harness would have caught a real leak had one been there.

05 — Limitations & honest framing

What this can't claim

The boldest sentence this project could have shipped was: "Random Forest is the clear winner on every metric — it correctly flags 95% of the top 20 highest-risk pages." That 95% figure is real, and it's also the naive-split number — exactly the one Section 4 shows was inflated. The rewritten, safe version: on a held-out set of client accounts the model never trained on, the ranking showed p@20 of 65% against a 39% base rate — an observed, directional improvement over the rule baseline on this split, meaningfully lower than the naive-split number would have suggested.

Three things this paper does not claim: it does not claim the ranking generalizes past these 32 clients — that's simply not measured here. It does not claim any causal relationship between a feature and decline — every number above is an association observed in one sample, under one split. And it does not claim to predict Google's ranking algorithm; the label is FlyRank's own operational definition of "declining," not a ground truth about search behavior.

The dataset itself is a bundled, anonymized starter slice — a fixed 30,000-row snapshot, not a live or continuously refreshed feed, and small enough (32 clients) that a handful of unusual accounts can move a metric noticeably.

06 — Ranked recommendations

What a reviewer should actually do with this

  1. Treat the queue as an aid, not a verdict. Use it to order attention, not to auto-publish changes — the safest first production use is inspecting the high-confidence rows and comparing the model's reasons against editorial judgment before acting.
  2. Start at high-confidence, not top-rank alone. Only 3,605 of 30,000 pages land in the high-confidence tier; that's a more honest starting queue than the raw top-N by score.
  3. Read the reason codes, not just the score. Every ranked page carries specific reason codes (e.g. declining_with_demand, low_ctr_visible_page) — they're what makes a flag checkable by a human in seconds, not a black-box number to trust blindly.
  4. Re-validate before trusting a bigger release. This holds for 30,000 rows and 32 clients; before leaning on it against the full ~79M-row warehouse, re-run the same client-grouped validation at that scale rather than assuming the numbers hold.
